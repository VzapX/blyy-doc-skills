# 运营约定 — 进度通报、上下文保护、异常容错、文件过滤

> **何时读取本文件**：执行 Phase 1/1.5/2/2A-2D 的子代理调度时。本文件汇总了贯穿所有阶段的运营性规范。
>
> SKILL.md 中只保留入口指引；具体执行细节以本文件为准。

## 一、进度通报规范

为避免用户在漫长的初始化过程中感到焦虑，**必须**在以下关键节点向用户输出进度信息。

### Phase 级进度（主 agent 输出）

| 时机 | 输出格式 |
|------|----------|
| 进入每个 Phase | `📋 Phase N — {阶段名} 开始...` |
| Phase 0 完成 | 汇报项目规模、执行模式（标准/大型）、检测到的 AI 工具 |
| Phase 0.1 /init 执行结果 | `✅ 已执行 /init，生成 CLAUDE.md`（或 `ℹ️ 当前工具不支持 /init，已跳过`） |
| Phase 1 旧文档迁移完成 | `📦 已迁移 N 个旧文档到 docs-old/`（或 `ℹ️ 无已有文档，跳过迁移`） |
| Phase 1 骨架创建完成 | `📁 已创建 N 个文档骨架文件` |
| Phase 1.5 清点完成 | `📦 发现 N 个旧文档文件，共约 M 行内容，已分为 K 个类别` |
| Phase 1.5 每个提取子代理启动 | `🔄 正在提取旧文档: {类别名}（{文件数} 个文件）...` |
| Phase 1.5 每个提取子代理完成 | `✅ {类别名} 提取完成：发现 N 个条目` |
| Phase 1.5 交叉校验完成 | `✅ 旧文档提取完成：X 个模块、Y 个业务术语、Z 个用户功能、W 个配置项` |
| 确定性清单预扫描完成 | `📊 项目清点完成（确定性扫描）: {N} 个实体、{M} 个控制器、{K} 个服务、{L} 个配置...` |
| Phase 2 每个子代理启动 | `🔄 正在分析: {任务名}（清单: {N} 个文件）...` |
| Phase 2 每个子代理完成 | `✅ {任务名} 分析完成（清单覆盖: N/N）` |
| 填充前审查关卡 — 清单覆盖率检查 | `📊 清单覆盖率: 实体 N/N ✓, 控制器 N/N ✓, 服务 N/M ✗ ...` |
| 填充前审查关卡 — T3 澄清开始 | `📋 填充前审查: 发现 {N} 个 T3 推测项需确认` |
| Phase 2 主 agent 整合每个文档 | `📝 正在填充 ({当前}/{总数}): {文档名}...` |
| Phase 3 完成 | 呈现最终完成报告（含完整性校验 + 可信度统计） |

### 子代理内部进度（子代理输出，缩进格式区分）

| 时机 | 输出格式 |
|------|----------|
| 每处理完 5 个文件 | `  ├─ [{任务名}] 已分析 5/{总数} 个{类型}文件: file1, file2, file3, file4, file5` |
| 每处理完 10 个文件 | `  ├─ [{任务名}] 已分析 10/{总数}: ...` |
| 每个分析子类别完成 | `  ├─ [{任务名}] {子类别}扫描完成: {已覆盖}/{清单总数} ✓` |
| 列表型内容收集中 | `  ├─ [{任务名}] 已收集 {N} 个{类型}（来自 {已处理}/{总数} 个文件）` |
| 子代理完成 | 结构化摘要：各类别条目数 + 清单覆盖率 + T3 条目数 |

**子代理完成摘要格式：**

```
✅ [{模块名/任务名}] 分析完成:
   - 实体/模型: 23 个 (清单覆盖: 23/23 ✓)
   - 服务: 18 个 (清单覆盖: 18/18 ✓)
   - 控制器: 12 个
   - 配置项: 8 个
   - T3 待澄清项: 3 个
```

> **原则**：宁可多输出几行进度，也不要让用户长时间看不到任何反馈。每完成一个有意义的步骤都应告知用户当前状态。子代理使用缩进（`├─` 前缀）与主 agent 的 Phase 级进度在视觉上区分。

## 二、上下文保护：边读边落盘策略

AI 工具的上下文窗口有限，大量信息一次性读入会导致早期内容被压缩丢弃。本 Skill 在所有阶段遵循**「边读边落盘」**原则，确保信息不因上下文溢出而丢失。

### 核心规则

1. **子代理隔离上下文**：每个子代理拥有独立上下文窗口，仅加载自己负责的文件范围。这是防止上下文溢出的第一道防线 — 充分利用子代理并行，将大量读取工作分散到多个独立上下文中
2. **即时落盘**：子代理完成分析后，**必须立即将结果写入文件**（标准模式写入 `docs/.init-temp/`，大型项目模式写入 `.init-docs/`），而非仅返回到主 agent 上下文中
3. **主 agent 只读摘要**：主 agent 在整合阶段，对每个子代理的落盘产物**只读取摘要/统计信息和条目列表**，不重新读取原始源文件
4. **分批处理**：当单个子代理的输入文件总量超过预估上下文安全阈值（约 800 行）时，该子代理应分批处理：读一批 → 提取 → 追加写入落盘文件 → 释放上下文 → 读下一批

### 各阶段应用

| 阶段 | 落盘策略 |
|------|---------|
| Phase 1.5 旧文档提取 | 每个提取子代理将结果即时写入 `extraction-{类别}.md` 文件。若某类别文件过多，分批读取并追加写入 |
| Phase 2 代码分析（标准模式） | 每个子代理完成后将分析素材写入 `docs/.init-temp/analysis-{任务名}.md`，主 agent 从文件读取而非从上下文接收 |
| Phase 2C 模块分析（大型模式） | 每个模块子代理将报告写入 `.init-docs/modules/{name}-analysis.md`（已有此设计，确保严格执行） |
| Phase 2/2D 文档填充 | 主 agent 每填充完一个文档立即写入磁盘，然后再开始下一个文档的填充。避免在上下文中同时持有多个文档的完整内容 |

### 标准模式临时文件清理

标准模式下产生的 `docs/.init-temp/` 目录，在 Phase 3 完成后自动清理。

## 三、异常处理与容错策略

### 子代理超时与失败

| 场景 | 处理策略 |
|------|---------|
| 子代理长时间无响应 | 主 agent 等待合理时间后，将该任务标记为失败，继续处理其他子代理结果。最终报告中标注受影响的文档 |
| 子代理输出格式异常 | 主 agent 尝试从输出中提取可用信息；若完全不可用，重新分发一次该任务 |
| 子代理提取结果明显不完整 | Phase 1.5 Step 3 的交叉校验已覆盖此场景 — 差异 > 20% 时自动重新分发 |

### 模式自动升级（标准 → 大型）

若标准模式执行过程中出现以下情况，**建议向用户提议切换到大型项目模式**：
- Phase 2 子代理数量超过 8 个（模块过多）
- 单个子代理分析的文件超过 200 个
- 上下文窗口频繁触发压缩警告

切换方式：暂停当前 Phase 2，将已完成的子代理产出迁移到 `.init-docs/` 目录，从 Phase 2A 的对应步骤恢复执行。

### Phase 1.5 提取失败的 fallback

若 Phase 1.5 提取完全失败（如旧文档格式无法解析）：
1. 向用户报告失败原因
2. 将 `docs-old/` 目录路径记录在完成报告中，提示用户手动参考
3. Phase 2 继续执行，仅基于代码扫描填充（无旧文档输入）
4. 完成报告中标注「旧文档未成功提取，建议人工核对 docs-old/ 中的内容」

## 四、文件过滤规则

所有扫描阶段应**默认排除**以下文件/目录：

```
# 依赖与构建产物
node_modules/, bin/, obj/, dist/, build/, out/, target/, packages/

# 版本控制
.git/, .svn/, .hg/

# IDE 配置
.vs/, .vscode/, .idea/, *.user, *.suo

# 生成代码（多语言）
*.Designer.cs, *.g.cs, *.generated.*     # C#
auto-generated/, __pycache__/, *.pyc      # Python
.next/, .nuxt/, .output/                  # 前端 SSR

# 二进制与媒体
*.dll, *.exe, *.pdb, *.jar, *.class
*.jpg, *.png, *.gif, *.ico, *.svg, *.woff, *.ttf

# 测试数据
**/testdata/, **/fixtures/, **/snapshots/, **/mocks/

# 历史迁移（仅读最新 3-5 个 + InitialCreate）
Migrations/*

# 锁文件
package-lock.json, yarn.lock, pnpm-lock.yaml, Gemfile.lock, poetry.lock
```
